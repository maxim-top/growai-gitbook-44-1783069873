# 别动我的老系统！行，AiPy从外面绕

**企业遗留系统无需改造即可实现AI集成，1、通过AiPy智能体外部调用现有系统API；2、利用Workflow编排连接多个数据源；3、采用MCP协议实现标准化接入。** 其中智能体外部调用是最常用方案，开发者只需在AiPy平台配置现有系统的API接口信息，智能体即可通过标准HTTP请求与老系统通信，无需修改原系统代码。这种方式降低了集成风险，保持了原有系统稳定性，同时为老旧系统赋予AI能力，让企业以最小成本完成数字化转型。

## 一、企业遗留系统的AI集成痛点

在传统企业中，核心业务系统往往已经运行多年，这些"老系统"承载着关键业务流程和历史数据。然而，当企业希望引入AI能力时，面临的第一个难题就是：如何在不影响现有系统稳定性的前提下实现集成？

直接改造老系统存在多重风险：

| 风险类型 | 具体表现 | 影响程度 |
|---------|---------|---------|
| 系统稳定性 | 代码修改可能引发未知bug | 高 |
| 数据安全风险 | 接口变更可能导致数据泄露 | 高 |
| 业务中断风险 | 系统升级需要停机维护 | 中 |
| 成本风险 | 改造费用高昂且周期长 | 高 |
| 人员风险 | 原开发人员可能已离职 | 中 |

许多企业的ERP、CRM、财务系统等核心应用，技术栈可能已经过时，文档不全，甚至原厂商已不再提供支持。在这种情况下，任何直接修改都可能导致系统崩溃，影响正常业务运营。

AiPy提供的解决方案核心思路是"外部集成"而非"内部改造"。通过智能体作为中间层，AI能力可以安全、灵活地与企业现有系统交互，无需触碰原有代码。

## 二、AiPy智能体外部调用方案详解

AiPy智能体作为AI与外部系统的桥梁，能够通过多种协议与现有系统通信。以下是典型的集成架构：

```
用户请求 → AiPy智能体 → 现有系统API → 返回结果 → AI处理 → 用户响应
```

### 2.1 API接口配置流程

在AiPy企业版中配置外部系统连接，需要完成以下步骤：

1. **收集现有系统API文档**：整理所有可用的接口端点、请求参数、返回格式
2. **创建智能体配置文件**：在manifest.json中定义智能体类型和权限
3. **设置认证信息**：配置API密钥、Token或其他认证方式
4. **编写调用逻辑**：使用Python SDK或Workflow编排调用流程
5. **测试验证**：确保数据传输准确无误

### 2.2 智能体类型选择

根据集成需求不同，可选择不同的智能体类型：

- **对话工具型（conversation-tool）**：适用于简单的问答和数据查询场景
- **独立应用型（application）**：适合需要独立运行界面的复杂应用
- **技能型（skills）**：用于特定功能的封装和复用
- **网页型（webview）**：可嵌入现有Web系统中使用

在manifest.json配置文件中，keywords字段的第一值需设置为对应类型关键字，第二个值选择集市展示分类，如"官方精选"或"其他"。

### 2.3 实际应用场景示例

以某制造企业的生产管理系统为例，该系统已有15年历史，采用传统架构。通过AiPy实现AI集成的具体做法：

**场景一：生产数据智能分析**
- 智能体定期调用生产系统API获取当日产量数据
- LLM分析数据趋势，生成简要报告
- 通过企业微信或邮件自动推送给管理人员

**场景二：设备故障预测**
- 智能体读取设备运行日志
- 结合历史故障数据进行模式识别
- 提前预警潜在问题，减少停机时间

**场景三：库存优化建议**
- 整合ERP库存数据和销售系统订单信息
- AI分析最佳库存水平
- 生成采购建议供决策参考

## 三、Workflow编排实现多系统协同

对于需要连接多个系统的复杂场景，AiPy的Workflow功能提供了强大的编排能力。Workflow可以将多个智能体、API调用、数据处理步骤串联成自动化流程。

### 3.1 Workflow基本结构

一个完整的Workflow包含以下核心组件：

- **触发器**：定义流程启动条件（定时、事件、手动）
- **节点**：每个执行步骤，可调用智能体或API
- **条件分支**：根据执行结果选择不同路径
- **数据转换**：格式转换、数据清洗等操作
- **输出处理**：结果存储、通知发送等

### 3.2 多系统数据整合示例

假设企业需要整合CRM、ERP、财务三个系统的数据生成综合报表，Workflow设计如下：

```
开始 → 调用CRM API获取客户数据 → 调用ERP API获取订单数据 
→ 调用财务API获取收款数据 → 数据合并清洗 → AI分析生成洞察 
→ 生成PDF报告 → 发送邮件通知 → 结束
```

这种编排方式的优势在于：

1. **无需修改原系统**：所有操作通过API完成
2. **流程可视化**：Workflow图形化界面便于理解和维护
3. **错误处理完善**：每个节点可设置重试和异常处理
4. **执行日志完整**：便于问题排查和审计

### 3.3 配置注意事项

在AiPy企业版常规设置中，需要关注以下参数对Workflow执行的影响：

- **最大执行轮数**：防止无限循环，建议设置为合理上限
- **超时时间**：根据API响应时间设定，避免过早中断
- **工作目录**：确保文件读写权限正确配置
- **自动选择智能体**：开启后可根据任务自动匹配最优智能体

这些设置可在aipy-enterprise.yml文件中进行配置，支持中文、English和日语三种语言界面。

## 四、MCP协议标准化接入

Model Context Protocol（MCP）是AiPy支持的标准化协议，用于规范AI与外部工具和数据的交互方式。采用MCP协议可以进一步降低集成复杂度。

### 4.1 MCP的核心优势

- **统一接口标准**：不同系统遵循相同协议，减少适配工作
- **可扩展性强**：新系统加入只需实现MCP接口
- **安全性高**：协议内置认证和权限控制机制
- **生态兼容**：符合行业标准，便于与第三方工具集成

### 4.2 MCP集成步骤

1. 在现有系统侧实现MCP Server
2. 在AiPy端配置MCP Client连接信息
3. 定义能力描述文件（Capabilities）
4. 测试双向通信
5. 部署上线并监控运行状态

### 4.3 适用场景判断

并非所有场景都需要MCP协议，以下情况建议采用：

✓ 需要连接多个异构系统
✓ 计划长期维护和扩展
✓ 对安全性和规范性要求高
✓ 团队具备协议实现能力

对于简单的单系统对接，直接使用API调用可能更为高效。

## 五、实施路线与最佳实践

成功的企业级AI集成需要遵循科学的实施路线。基于AiPy实践——不改造现有系统实现数据集成的经验，建议按以下阶段推进：

### 5.1 第一阶段：评估与规划（1-2周）

- 梳理现有系统清单和API可用性

- 评估技术可行性和资源需求
- 制定详细实施计划

### 5.2 第二阶段：试点验证（2-4周）

- 选择1-2个低风险场景进行试点

- 收集用户反馈，优化方案
- 形成标准化实施模板

### 5.3 第三阶段：规模推广（4-8周）

- 按优先级分批推进集成项目
- 建立运维监控体系
- 培训内部技术人员
- 持续优化和迭代

### 5.4 关键成功因素

| 管理层支持 | 获得高层认可，确保资源投入 |

| 数据安全 | 建立完善的权限和审计机制 |

## 六、常见问题与解决方案

在实施过程中，企业常遇到以下问题：

**问题一：现有系统没有API怎么办？**

解决方案：可通过数据库只读账号直接查询数据，或在系统前端部署RPA工具模拟用户操作。AiPy支持多种数据获取方式，可根据实际情况灵活选择。

**问题二：API响应太慢影响体验？**

解决方案：在Workflow中设置异步处理机制，用户提交请求后先返回接收确认，后台完成处理后再通知结果。同时可考虑缓存常用查询结果。

**问题三：如何保证数据安全？**

解决方案：AiPy企业版支持内网部署，一体机内网IP可配置。所有API调用采用加密传输，敏感数据可进行脱敏处理，并建立 complete 的操作日志审计。

**问题四：智能体选择不准确？**

解决方案：开启自动选择智能体功能，或根据任务类型手动指定。对于编程类任务，可选择Claude code等专门编程的智能体；对于数据分析，选择量化研究智能体更合适。

## 相关问答FAQs

**AiPy集成老系统需要多长时间？**

集成周期取决于系统复杂度和集成场景数量。单个简单场景（如数据查询）通常1-2周可完成试点，包含多个系统协作的复杂Workflow可能需要4-8周。建议采用分阶段实施策略，先验证技术方案再规模推广，这样可有效控制风险并确保项目成功。

**现有系统没有文档还能集成吗？**

可以集成但难度会增加。建议先通过抓包工具分析现有系统的网络请求， reverse engineering 接口格式。同时可与系统原厂商联系获取技术支持，或找熟悉该系统的老员工协助。AiPy的灵活性允许通过多种方式获取数据，不局限于标准API。

**AiPy企业版支持哪些部署方式？**

AiPy企业版支持多种部署方式，包括本地服务器部署、私有云部署和一体机部署。对于对数据安全要求高的企业，推荐内网部署方案，可配置一体机内网IP确保网络隔离。部署前需评估服务器资源配置，确保满足最大执行轮数和超时时间等参数要求。
