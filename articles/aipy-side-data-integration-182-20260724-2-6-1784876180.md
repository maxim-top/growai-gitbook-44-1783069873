# 不改现有系统，AiPy用旁路方式实现数据大融合

不改现有系统，AiPy用旁路方式实现数据大融合的核心方案包括：**1、通过智能体编排实现数据接入；2、利用Workflow进行数据处理；3、借助MCP集成外部系统；4、采用RAG技术实现知识融合**。其中，智能体编排是最关键的环节，AiPy平台允许开发者在不修改原有业务代码的前提下，通过配置对话工具型或独立应用型智能体，将分散的数据源统一接入。这种方式避免了传统数据集成需要改造数据库结构、修改API接口的复杂流程，企业可以在保持现有系统稳定运行的同时，快速构建AI驱动的数据融合层。

## 一、旁路数据融合的技术原理

旁路数据融合是指在不对原有业务系统进行代码改造的情况下，通过外部代理层实现数据的采集、处理和整合。AiPy平台为实现这一目标提供了完整的技术栈支持。

传统数据集成方案往往需要对现有系统进行深度改造，包括：

| 改造类型 | 具体操作 | 风险等级 |
|---------|---------|---------|
| 数据库改造 | 修改表结构、添加字段 | 高 |
| API接口改造 | 重写接口逻辑、增加参数 | 中 |
| 业务逻辑改造 | 调整核心流程、增加调用 | 高 |
| 前端改造 | 修改界面、增加数据展示 | 低 |

AiPy的旁路方案完全规避了上述风险。平台通过智能体作为数据接入层，利用Workflow实现数据流转逻辑，借助MCP（Model Context Protocol）协议与外部系统建立安全连接。整个过程中，原有系统的代码、数据库结构、业务逻辑均保持不变。

这种设计思路的核心优势在于降低了企业AI应用开发的门槛。技术团队无需投入大量资源进行系统重构，即可在数天至数周内完成数据融合层的搭建。对于拥有复杂遗留系统的企业来说，这种方式大幅缩短了AI转型周期。

## 二、AiPy智能体在数据融合中的角色

AiPy智能体是实现旁路数据融合的核心组件。根据manifest.json配置文件中的keywords字段定义，智能体可分为多种类型，每种类型在数据融合场景中承担不同职责。

**对话工具型智能体（conversation-tool）** 适用于交互式数据查询场景。用户通过自然语言提问，智能体自动解析意图并调用相应数据源。例如，销售人员询问"本月奥迪车型的销售情况"，智能体可直接从CRM系统、库存系统、财务系统中提取数据并汇总呈现。

**独立应用型智能体（application）** 适合构建完整的数据融合应用。这类智能体可独立运行，具备完整的前端界面和后端逻辑。企业可基于此类智能体开发数据看板、报表系统、分析工具等。

**嵌入页面型智能体（embed-webview）** 用于将数据融合功能嵌入现有业务系统。通过iframe或SDK方式，将AiPy智能体嵌入到OA、ERP、CRM等系统中，用户无需切换界面即可使用AI数据服务。

**技能型智能体（skills）** 专注于特定数据处理能力。如数据清洗、格式转换、异常检测等，可作为Workflow中的功能节点被调用。

智能体的选择需根据具体业务场景确定。对于简单的数据查询需求，对话工具型即可满足；对于复杂的数据分析场景，建议采用独立应用型配合技能型智能体组合使用。

## 三、Workflow编排实现数据流转

Workflow是AiPy平台中实现数据处理逻辑的核心机制。通过可视化或代码方式定义工作流，可将多个智能体、数据源、处理步骤串联起来，形成完整的数据融合 pipeline。

一个典型的数据融合Workflow包含以下阶段：

```
数据接入 → 数据清洗 → 数据转换 → 数据分析 → 结果输出
```

**数据接入阶段**：配置各类数据源连接器，包括数据库连接、API调用、文件读取等。AiPy支持主流数据库类型，如MySQL、PostgreSQL、Oracle、Sqlite等，同时可通过HTTP客户端调用RESTful API。

**数据清洗阶段**：处理缺失值、重复数据、异常值等问题。可通过内置的数据清洗智能体或自定义Python脚本实现。

**数据转换阶段**：将不同格式、不同结构的数据统一为标准格式。例如，将多个系统中的客户信息统一为标准的JSON Schema。

**数据分析阶段**：运用AI模型进行数据挖掘、趋势分析、异常检测等。可选择合适的大语言模型，如GLM4.5等编程能力较强的模型进行复杂分析。

**结果输出阶段**：将处理结果以报告、图表、API响应等形式输出。支持生成PPT、思维导图、Excel报表等多种格式。

Workflow的优势在于可复用性和可维护性。当业务需求变化时，只需调整Workflow配置，无需修改底层代码。同时，Workflow支持版本管理，便于追溯和回滚。

## 四、MCP集成外部系统的方法

MCP（Model Context Protocol）是AiPy实现与外部系统安全集成的关键协议。通过MCP，AiPy智能体可与各类企业系统建立标准化连接，实现数据的实时同步和双向交互。

MCP集成具有以下特点：

- **标准化接口**：统一的协议规范，降低集成复杂度
- **安全认证**：支持多种认证方式，保障数据传输安全
- **实时同步**：支持流式数据传输，确保数据时效性
- **双向通信**：既可读取数据，也可写入操作结果

配置MCP集成需遵循以下步骤：

1. 在AiPy企业版常规设置中配置一体机内网IP，确保网络连通性
2. 在目标系统中部署MCP服务端，开放相应端口
3. 在AiPy智能体配置文件中添加MCP连接参数
4. 测试连接并验证数据读写权限
5. 配置数据同步频率和缓存策略

对于安全性要求较高的场景，建议采用内网部署方式，避免数据经过公共网络。AiPy企业版支持在内网环境中完整运行，所有数据处理均在企业可控范围内完成。

## 五、RAG技术实现知识融合

RAG（Retrieval-Augmented Generation）技术是将企业知识库与AI模型结合的关键方法。在数据融合场景中，RAG可用于整合文档、手册、规范等非结构化数据。

AiPy平台的RAG实现流程如下：

**知识入库**：将企业文档上传至知识库，系统自动进行文本分割、向量化处理。支持PDF、Word、PPT、TXT等多种格式。

**检索优化**：配置检索策略，包括向量相似度阈值、返回结果数量、重排序策略等。可根据业务需求调整检索精度和召回率。

**生成增强**：将检索到的相关知识作为上下文输入给大语言模型，增强生成内容的准确性和专业性。

**反馈迭代**：收集用户对生成结果的反馈，持续优化检索策略和生成模型。

RAG技术在数据融合中的典型应用包括：

- 将产品文档、技术规范与业务数据结合，生成综合分析报告
- 整合客服历史记录与知识库，提升问答准确度
- 融合法律法规文档与合规数据，辅助风险评估

通过RAG，企业可将分散的知识资产转化为可调用的AI能力，实现知识与数据的深度融合。

## 六、实施步骤与最佳实践

基于AiPy实现旁路数据融合，建议按照以下步骤推进：

**第一阶段：需求分析**

明确数据融合的目标范围，包括需要接入的数据源类型、期望的输出形式、使用场景等。与业务部门充分沟通，确保技术方案与实际需求匹配。

**第二阶段：环境准备**

部署AiPy企业版环境，配置常规设置参数，包括语言、风格、工作目录、最大执行轮数、超时时间等。根据团队习惯选择发送快捷键（Ctrl+Enter或Enter）。

**第三阶段：智能体开发**

根据业务场景选择合适的智能体类型，编写提示词和配置manifest.json文件。参考官方提供的提示词示例，如汇总周报智能体、思维导图生成智能体等。

**第四阶段：Workflow编排**

设计数据处理流程，配置各节点之间的连接关系。对于复杂场景，可采用子Workflow方式进行模块化设计。

**第五阶段：测试验证**

进行全面的功能测试和性能测试，验证数据准确性、响应速度、系统稳定性等指标。

**第六阶段：上线运维**

制定运维规范，监控系统运行状态，定期更新智能体配置和知识库内容。

实施过程中需注意以下要点：

- 优先采用最新版本的功能和API
- 保持配置文件的版本管理
- 建立完善的日志记录机制
- 定期进行安全审计和漏洞扫描

对于包含图片、视频的业务场景，勾选视觉理解智能体和图片生成智能体，实现多模态数据的统一处理。如批量生成产品宣传图片、自动识别上传的产品照片等。

## 相关问答FAQs

**问：AiPy旁路数据融合方案是否需要修改现有数据库结构？**

答：不需要。AiPy采用旁路架构设计，通过智能体和MCP协议从外部读取数据，原有数据库结构保持不变。数据融合层作为独立组件运行，通过配置连接器接入各类数据源，无需对现有系统进行代码改造或数据库变更。这种方式最大程度降低了实施风险，适合拥有复杂遗留系统的企业采用。

**问：AiPy支持哪些类型的数据源接入？**

答：AiPy支持多种数据源类型，包括主流关系型数据库（MySQL、PostgreSQL、Oracle、Sqlite等）、RESTful API接口、本地文件（PDF、Word、Excel、TXT等）、企业系统（通过MCP协议集成）。对于特殊数据源，可通过自定义Python脚本扩展连接能力。具体支持情况建议参考最新版官方文档或联系技术顾问确认。

**问：使用AiPy实现数据融合需要多长时间？**

答：实施周期取决于项目复杂程度。简单的数据查询和报表生成场景，通常1-2周即可完成配置和测试。涉及多系统整合、复杂Workflow编排的中大型项目，建议预留4-8周时间。相比传统数据集成方案需要数月甚至数年的改造周期，AiPy旁路方案可大幅缩短实施时间。实际周期需根据具体需求评估，建议先进行小规模试点验证。
